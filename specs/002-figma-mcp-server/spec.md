# Feature Specification: Figma MCP Server for Claude Code

**Feature Branch**: `002-figma-mcp-server`
**Created**: 2026-03-02
**Status**: Draft
**Input**: MCP-сервер на основе figma-scaler с фичами из cursor-talk-to-figma для пиксель-перфект вёрстки в Claude Code

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Получение дизайн-токенов через MCP (Priority: P1)

Разработчик подключает MCP-сервер к Claude Code. При вёрстке
компонента Claude Code вызывает инструмент `get_design_tokens`
с указанием Figma-файла и получает полный набор дизайн-токенов
(цвета, типографика, отступы, радиусы, тени, градиенты) в
структурированном формате. Токены кэшируются между вызовами —
повторные запросы к тому же файлу возвращают результат мгновенно.

**Why this priority**: Без токенов Claude Code не может верстать
pixel-perfect. Это фундамент: AI должен знать точные значения
цветов, шрифтов, отступов из дизайн-системы.

**Independent Test**: Подключить MCP-сервер к Claude Code,
вызвать `get_design_tokens` для реального Figma-файла, убедиться
что токены возвращаются и соответствуют ручной инспекции в Figma.

**Acceptance Scenarios**:

1. **Given** настроенный MCP-сервер и Figma file ID, **When**
   Claude Code вызывает `get_design_tokens`, **Then** система
   возвращает структурированный набор токенов (цвета, типографика,
   отступы, радиусы, тени, градиенты) со значениями и CSS-переменными.
2. **Given** ранее запрошенный файл, **When** Claude Code повторно
   вызывает `get_design_tokens` для того же файла, **Then** результат
   возвращается из кэша без обращения к Figma API.
3. **Given** невалидный file ID или токен, **When** вызов
   `get_design_tokens`, **Then** система возвращает понятное сообщение
   об ошибке (не stack trace).

### User Story 2 — Инспекция конкретной ноды или компонента (Priority: P1)

Разработчик верстает конкретный компонент. Claude Code вызывает
`get_node_info` с node ID и получает полную информацию о ноде:
размеры, позиция, auto-layout настройки, заливки, обводки, тени,
типографика, дочерние элементы с иерархией. Данные достаточны
для полного воспроизведения компонента в HTML/CSS.

**Why this priority**: Токены — это словарь, а информация о ноде —
это конкретное "предложение". Для pixel-perfect вёрстки нужно знать
точную структуру и свойства каждого элемента.

**Independent Test**: Вызвать `get_node_info` для компонента из Figma,
по полученным данным воспроизвести его в HTML/CSS — результат должен
визуально совпадать с оригиналом.

**Acceptance Scenarios**:

1. **Given** валидный node ID компонента с auto-layout, **When**
   Claude Code вызывает `get_node_info`, **Then** результат содержит:
   размеры, layoutMode, padding, itemSpacing, alignment, заливки,
   обводки, тени, cornerRadius, и полную иерархию дочерних нод.
2. **Given** node ID текстовой ноды, **When** вызов `get_node_info`,
   **Then** результат содержит: текстовое содержимое, fontFamily,
   fontSize, fontWeight, lineHeight, letterSpacing, цвет текста,
   textAlign.
3. **Given** node ID COMPONENT_SET с вариантами, **When** вызов
   `get_node_info`, **Then** результат содержит все варианты с
   полными свойствами и метками (State=Default/Hover, Size=S/M/L).

### User Story 3 — Получение CSS Custom Properties (Priority: P2)

Разработчик хочет подключить дизайн-систему к проекту. Claude Code
вызывает `get_css_variables` и получает готовый CSS-файл с Custom
Properties, сгруппированными по категориям. Файл можно напрямую
использовать в проекте.

**Why this priority**: CSS-переменные — это мост между токенами
и кодом. Без них Claude Code будет хардкодить значения вместо
использования переменных.

**Independent Test**: Вызвать `get_css_variables`, подключить
полученный CSS к проекту, убедиться что все переменные валидны
и значения совпадают с Figma.

**Acceptance Scenarios**:

1. **Given** Figma-файл с токенами, **When** Claude Code вызывает
   `get_css_variables`, **Then** возвращается валидный CSS с Custom
   Properties, сгруппированными по категориям (цвета, типографика,
   spacing, radii, тени).
2. **Given** CSS-переменные уже получены, **When** Claude Code
   пишет стили для компонента, **Then** он использует
   `var(--color-xxx)`, `var(--spacing-xxx)` и т.д. вместо хардкода.

### User Story 4 — Экспорт изображений нод (Priority: P2)

Разработчик верстает страницу с иконками и иллюстрациями.
Claude Code вызывает `export_node_image` с node ID и форматом
(SVG/PNG) и получает изображение, сохранённое в указанную
директорию проекта. Для иконок используется SVG, для растровых
изображений — PNG.

**Why this priority**: Pixel-perfect вёрстка включает не только
CSS, но и ассеты. Без экспорта изображений разработчик должен
вручную экспортировать каждую иконку из Figma.

**Independent Test**: Вызвать `export_node_image` для иконки (SVG)
и для фото (PNG), убедиться что файлы сохранены корректно и
визуально идентичны оригиналу в Figma.

**Acceptance Scenarios**:

1. **Given** node ID иконки, **When** Claude Code вызывает
   `export_node_image` с форматом SVG, **Then** SVG-файл сохраняется
   в указанную директорию и визуально совпадает с оригиналом.
2. **Given** node ID растрового изображения, **When** вызов с форматом
   PNG и scale=2, **Then** PNG-файл сохраняется в 2x разрешении.
3. **Given** несуществующий node ID, **When** вызов `export_node_image`,
   **Then** система возвращает понятное сообщение об ошибке.

---

### User Story 5 — Обзор структуры документа (Priority: P2)

Разработчик начинает работу с незнакомым Figma-файлом. Claude Code
вызывает `get_document_structure` и получает обзор: список страниц,
верхнеуровневые фреймы с именами и размерами, количество компонентов.
Это позволяет понять структуру файла и выбрать нужную секцию
для дальнейшей работы.

**Why this priority**: Перед детальным парсингом нужно понять общую
картину. Большие файлы содержат десятки страниц — разработчик
должен быстро найти нужную секцию.

**Independent Test**: Вызвать `get_document_structure` для реального
файла с несколькими страницами и компонентами. Результат должен
отражать реальную структуру документа.

**Acceptance Scenarios**:

1. **Given** Figma-файл с несколькими страницами, **When** Claude Code
   вызывает `get_document_structure`, **Then** результат содержит
   имя файла, список страниц с количеством дочерних элементов,
   и верхнеуровневые фреймы каждой страницы (имя, node ID, размеры).
2. **Given** файл с компонентами, **When** вызов `get_document_structure`,
   **Then** результат включает список компонентов и component sets
   с их node ID.

### User Story 6 — AI-контекст для вёрстки (Priority: P3)

Разработчик начинает сессию вёрстки. Claude Code вызывает
`get_design_context` и получает компактную сводку дизайн-системы,
оптимизированную для контекстного окна: топ-цвета по usage count,
шкалу отступов, типографические стили, правила использования токенов.
Claude Code запоминает эту сводку и использует её во всех
последующих запросах вёрстки.

**Why this priority**: Это "мультипликатор" — одна команда даёт
Claude Code полный контекст дизайн-системы, что повышает качество
всей последующей вёрстки.

**Independent Test**: Вызвать `get_design_context`, затем попросить
Claude Code сверстать компонент — он должен использовать правильные
CSS-переменные без дополнительных подсказок.

**Acceptance Scenarios**:

1. **Given** Figma-файл с извлечёнными токенами, **When** Claude Code
   вызывает `get_design_context`, **Then** результат содержит: топ-10
   цветов с usage count, все spacing-значения, типографические стили,
   правила использования (всегда var(), никогда хардкод).
2. **Given** полученный контекст, **When** Claude Code верстает
   компонент, **Then** он корректно использует CSS-переменные из
   дизайн-системы.

### User Story 7 — Поиск токенов по значению (Priority: P3)

Разработчик видит в макете цвет или размер шрифта и хочет найти
соответствующий токен. Claude Code вызывает `search_token` с
примерным значением (hex-цвет, число) и получает ближайшие
совпадения из дизайн-системы с именами CSS-переменных.

**Why this priority**: При вёрстке часто нужно найти "какой токен
соответствует этому цвету?" — поиск по значению экономит время
и предотвращает хардкод.

**Independent Test**: Вызвать `search_token` с hex-цветом из Figma,
убедиться что возвращается правильная CSS-переменная.

**Acceptance Scenarios**:

1. **Given** hex-цвет из Figma-макета, **When** Claude Code вызывает
   `search_token` с этим значением, **Then** система возвращает
   совпадающий токен (имя, CSS-переменная, usage count) или ближайшие
   по цвету варианты.
2. **Given** число (размер шрифта или отступ), **When** вызов
   `search_token`, **Then** система возвращает совпадающие токены
   из всех категорий (font-size, spacing, radius).

### Edge Cases

- Что происходит при обрыве соединения с Figma API во время запроса?
  Система MUST вернуть понятное сообщение об ошибке и предложить
  повторить запрос. Частичные данные НЕ возвращаются.
- Что происходит при запросе к очень большому файлу (10 000+ нод)?
  Система MUST кэшировать результат и при наличии `page` или
  `node` фильтра обрабатывать только указанную секцию.
- Как обрабатывается истекший кэш?
  Кэш MUST содержать timestamp и автоматически устаревать через
  30 минут (TTL). При запросе с флагом `force_refresh` кэш
  игнорируется принудительно. При истечении TTL следующий запрос
  прозрачно перезапрашивает данные из Figma API.
- Что если Figma API токен не настроен?
  Система MUST при первом вызове любого инструмента вернуть
  понятную инструкцию по настройке токена.
- Что происходит при rate limit (429) от Figma API?
  Система MUST ждать Retry-After секунд и повторить запрос
  (до 3 попыток), прозрачно для пользователя.
- Как обрабатываются параллельные запросы к одному файлу?
  Система MUST использовать один fetch для параллельных запросов
  к одному файлу (deduplication).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST реализовать MCP-протокол (Model Context
  Protocol) и подключаться к Claude Code как MCP-сервер.
- **FR-002**: Система MUST предоставлять инструмент `get_design_tokens`
  для извлечения полного набора токенов (цвета, градиенты, типографика,
  отступы, радиусы, тени, изображения, компоненты) из Figma-файла
  по его ID или URL. Поддерживается фильтрация по странице или node ID.
- **FR-003**: Система MUST предоставлять инструмент `get_node_info`
  для получения полной информации о конкретной ноде: размеры,
  позиция, auto-layout, заливки, обводки, тени, типографика,
  cornerRadius, и иерархия дочерних нод (до 5 уровней вложенности).
  Каждое визуальное свойство MUST содержать inline CSS-маппинг:
  поле `css_variable` (напр. `var(--color-brand-primary)`) и
  `css_property` (напр. `background-color`) рядом со значением,
  чтобы AI мог сразу использовать их в коде.
- **FR-004**: Система MUST предоставлять инструмент `get_css_variables`
  для генерации CSS Custom Properties файла из токенов, с
  группировкой по категориям.
- **FR-005**: Система MUST предоставлять инструмент `export_node_image`
  для экспорта любой ноды как изображения (SVG, PNG, JPG, PDF)
  через Figma Images API. Файл сохраняется в указанную директорию.
- **FR-006**: Система MUST предоставлять инструмент
  `get_document_structure` для получения обзора Figma-файла:
  страницы, верхнеуровневые фреймы, компоненты.
- **FR-007**: Система MUST предоставлять инструмент `get_design_context`
  для генерации компактной AI-оптимизированной сводки дизайн-системы
  с правилами использования токенов.
- **FR-008**: Система MUST предоставлять инструмент `search_token`
  для поиска токенов по значению (hex-цвет, число) с возвратом
  ближайших совпадений.
- **FR-009**: Система MUST кэшировать результаты запросов к Figma API.
  Кэш привязан к file_id и содержит timestamp. Повторные запросы
  к тому же файлу возвращают данные из кэша. Кэш автоматически
  устаревает через 30 минут (TTL). Кэш также обновляется
  принудительно по флагу `force_refresh`.
- **FR-010**: Система MUST принимать Figma API токен через
  конфигурацию MCP-сервера (переменная окружения или аргумент).
  Токен MUST NEVER включаться в ответы инструментов.
- **FR-011**: Система MUST обрабатывать HTTP 429 (rate limit)
  автоматически: ждать Retry-After, повторить до 3 раз, после чего
  возвращать ошибку.
- **FR-012**: Система MUST возвращать данные в структурированном
  JSON-формате, оптимизированном для обработки AI-ассистентом.
- **FR-013**: Система MUST переиспользовать существующую логику
  figma-scaler (экстракторы, pipeline, writers) без дублирования.
- **FR-014**: Система MUST предоставлять инструмент `get_nodes_info`
  для получения информации о нескольких нодах одним вызовом (batch).
- **FR-015**: Все инструменты, связанные с конкретным файлом, MUST
  принимать `file_id` как параметр. Если файл ранее кэширован —
  использовать кэш; если нет — запросить из API.
- **FR-016**: Система MUST предоставлять MCP-ресурс (resource)
  `figma://tokens/{file_id}` для автоматического доступа к токенам
  без явного вызова инструмента.
- **FR-017**: Система MUST предоставлять MCP Prompt `layout_strategy`
  с правилами pixel-perfect вёрстки: всегда вызывать
  `get_design_context` перед началом вёрстки, использовать
  `var(--xxx)` вместо хардкода, сверять значения с токенами.
- **FR-018**: Система MUST предоставлять MCP Prompt
  `read_design_strategy` с инструкциями по чтению дизайна:
  начинать с `get_document_structure`, затем `get_node_info`
  для целевых компонентов, использовать `search_token` для
  маппинга значений к CSS-переменным.
- **FR-019**: Система MUST предоставлять MCP Prompt
  `token_usage_rules` с правилами использования токенов:
  никогда не хардкодить цвета/отступы/шрифты, всегда
  ссылаться на CSS-переменные из дизайн-системы, при
  отсутствии подходящего токена — явно сообщать разработчику.

### Key Entities

- **MCPServer**: MCP-сервер, управляющий подключением к Claude Code,
  регистрацией инструментов и маршрутизацией запросов.
- **TokenCache**: Кэш извлечённых токенов и данных файла, привязанный
  к file_id и timestamp. Позволяет мгновенный доступ к ранее
  запрошенным данным.
- **DesignToken**: Обобщённый тип извлечённого токена (наследуется
  из figma-scaler): ColorToken, GradientToken, TypographyToken,
  SpacingToken, RadiusToken, ShadowToken, ImageToken.
- **NodeDetail**: Полная информация о ноде Figma: свойства, стили,
  layout-данные, дочерние элементы. Каждое визуальное свойство
  содержит inline CSS-маппинг (`css_variable` + `css_property`),
  позволяющий AI копировать значения напрямую в код без поиска.
- **DesignContext**: Компактная сводка дизайн-системы: топ-токены,
  шкалы, правила — оптимизирована для контекстного окна AI.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Claude Code MUST иметь возможность получить все
  дизайн-токены из Figma-файла одним вызовом инструмента,
  без ручного запуска CLI и чтения файлов.
- **SC-002**: Повторные запросы к ранее обработанному файлу MUST
  возвращать результат менее чем за 1 секунду (из кэша).
- **SC-003**: Информация о конкретной ноде MUST быть достаточной
  для воспроизведения её в HTML/CSS без обращения к Figma напрямую
  (включая размеры, layout, стили, типографику).
- **SC-004**: AI-ассистент, используя данные `get_node_info`,
  SHOULD воспроизводить все визуальные свойства компонента
  (цвета, размеры, отступы, типографика, тени) с использованием
  корректных CSS-переменных из дизайн-системы.
- **SC-005**: Настройка MCP-сервера в Claude Code MUST занимать
  не более 2 минут (одна команда или конфигурация).
- **SC-006**: Экспорт изображения ноды в SVG или PNG MUST
  завершаться менее чем за 10 секунд.
- **SC-007**: Ответы инструментов `get_node_info` и `get_nodes_info`
  MUST содержать inline CSS-маппинги (`css_variable` + `css_property`)
  для каждого визуального свойства. Ответ `search_token` MUST
  содержать `css_variable` для каждого совпадения. Это позволяет
  AI сразу использовать `var(--xxx)` в коде.

## Clarifications

### Session 2026-03-02

- Q: Нужно ли добавить MCP Prompts для направления AI при вёрстке? → A: Да, добавить 2-3 MCP Prompts: стратегия вёрстки, стратегия чтения дизайна, правила использования токенов.
- Q: Должен ли кэш автоматически устаревать (TTL)? → A: Да, TTL 30 минут — кэш автоматически устаревает, следующий запрос перезапрашивает данные.
- Q: Как маппить свойства ноды к CSS в ответе get_node_info? → A: Inline CSS hints — каждое свойство ноды содержит поле css_variable и css_property рядом со значением.

## Assumptions

- Figma API токен доступен через переменную окружения `$FIGMA_TOKEN`
  или передаётся в конфигурации MCP-сервера.
- Claude Code поддерживает MCP-серверы, запускаемые как stdio-процессы.
- Существующая кодовая база figma-scaler (экстракторы, pipeline,
  writers) используется как библиотека без модификации.
- MCP-сервер работает локально на машине разработчика (не удалённо).
- Figma REST API v1 используется для всех запросов (как в figma-scaler).
- Кэш хранится в памяти процесса (не на диске). При перезапуске
  сервера кэш сбрасывается.
