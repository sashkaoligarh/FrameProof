# Feature Specification: Figma Design Parser

**Feature Branch**: `001-figma-design-parser`
**Created**: 2026-02-27
**Status**: Draft
**Input**: Pixel-perfect парсинг элементов и страниц Figma для создания дизайн-системы и последующей вёрстки с AI-ассистентом

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Извлечение дизайн-токенов из Figma-файла (Priority: P1)

Разработчик указывает Figma-файл (по ID или URL) и получает полный
набор дизайн-токенов: цвета, типографика, отступы, радиусы,
тени, градиенты. Токены генерируются в формате CSS Custom Properties
и в JSON для программного использования. Результат — единый источник
правды, готовый к импорту в любой фронтенд-проект.

**Why this priority**: Без точных токенов невозможна pixel-perfect
вёрстка. Это фундамент, на котором строится всё остальное.

**Independent Test**: Запуск парсера на реальном Figma-файле и сравнение
полученных токенов с ручной инспекцией элементов в Figma — значения
MUST совпадать.

**Acceptance Scenarios**:

1. **Given** валидный Figma file ID и API-токен, **When** пользователь
   запускает парсер, **Then** система выгружает полный набор токенов
   (цвета, типографика, отступы, радиусы, тени, градиенты) в JSON
   и CSS файлы без потерь данных.
2. **Given** Figma-файл с компонентами, использующими внешнюю
   библиотеку цветов, **When** парсер обрабатывает файл, **Then**
   цвета из внешней библиотеки извлекаются корректно (значения,
   а не ссылки на стили).
3. **Given** невалидный file ID или просроченный токен, **When**
   пользователь запускает парсер, **Then** система выдаёт понятное
   сообщение об ошибке с указанием причины.

---

### User Story 2 — Извлечение структуры компонентов с layout-данными (Priority: P2)

Разработчик получает детальную информацию о каждом компоненте:
размеры, auto-layout настройки (direction, padding, gap, alignment),
вложенность, варианты (variants). Данные достаточны для воспроизведения
layout в CSS/HTML без обращения к Figma.

**Why this priority**: Токены дают значения, но без структуры
компонентов невозможно собрать pixel-perfect layout. Это второй
критический слой.

**Independent Test**: Взять 3-5 компонентов из Figma, сгенерировать
их описание парсером, и по этому описанию воссоздать layout —
результат MUST визуально совпадать с оригиналом.

**Acceptance Scenarios**:

1. **Given** компонент с auto-layout, **When** парсер обрабатывает его,
   **Then** результат содержит: layoutMode, direction, padding
   (top/right/bottom/left), itemSpacing, counterAxisSpacing,
   primaryAxisAlignItems, counterAxisAlignItems, layoutWrap.
2. **Given** COMPONENT_SET с вариантами (напр. State=Default/Hover,
   Size=S/M/L), **When** парсер обрабатывает его, **Then** каждый
   вариант представлен отдельной записью с полным набором свойств
   и меткой варианта.
3. **Given** компонент с глубокой вложенностью (5+ уровней), **When**
   парсер обрабатывает его, **Then** иерархия нод сохранена полностью
   с правильными parent-child связями.

---

### User Story 3 — Генерация AI-оптимизированного контекста (Priority: P3)

Система генерирует структурированный контекст-файл (Markdown или JSON),
оптимизированный для использования AI-ассистентом (Claude Code, Cursor
и т.д.) при вёрстке. Файл содержит: правила использования токенов,
топ-используемые значения, маппинг компонентов к ID, рекомендации
по импорту.

**Why this priority**: Качественный AI-контекст — это мультипликатор
эффективности. Точные токены + хорошая структура + правильный контекст
= AI может верстать pixel-perfect.

**Independent Test**: Подать сгенерированный контекст-файл AI-ассистенту
вместе с скриншотом Figma-компонента — AI SHOULD воспроизвести
компонент без обращения к Figma напрямую.

**Acceptance Scenarios**:

1. **Given** успешно извлечённые токены и компоненты, **When** парсер
   генерирует контекст, **Then** контекст-файл содержит: таблицу
   цветов с usage count, таблицу типографики, spacing scale, маппинг
   компонентов к Figma node ID.
2. **Given** контекст-файл, **When** AI-ассистент читает его, **Then**
   AI может корректно использовать CSS-переменные из дизайн-системы
   без хардкода значений.

---

### User Story 4 — Парсинг конкретной страницы или секции (Priority: P3)

Разработчик может указать конкретную страницу Figma-файла или
конкретный node ID для парсинга, вместо обработки всего файла целиком.
Это ускоряет работу с большими файлами и позволяет фокусироваться
на нужном разделе.

**Why this priority**: Большие Figma-файлы могут содержать сотни
страниц. Полный парсинг — медленный и избыточный для повседневной
работы.

**Independent Test**: Указать node ID конкретного фрейма и проверить,
что парсер обработал только его содержимое.

**Acceptance Scenarios**:

1. **Given** Figma-файл с несколькими страницами, **When** пользователь
   указывает имя или ID конкретной страницы, **Then** парсер
   обрабатывает только эту страницу.
2. **Given** конкретный node ID фрейма, **When** пользователь передаёт
   его парсеру, **Then** парсер обрабатывает только содержимое этого
   фрейма и его дочерних нод.

---

### Edge Cases

- Что происходит при пустом Figma-файле (нет нод)?
  Система MUST вернуть пустые токены с предупреждением, а не упасть.
- Что происходит при слишком большом файле (10 000+ нод)?
  Система MUST обработать файл, показывая прогресс, и при необходимости
  предложить парсинг по страницам.
- Как обрабатываются скрытые слои (visible: false)?
  По умолчанию скрытые слои MUST пропускаться, но с флагом
  `--include-hidden` MUST включаться.
- Как обрабатываются ноды с нулевой непрозрачностью (opacity: 0)?
  MUST пропускаться при извлечении цветов (порог непрозрачности 0.02),
  но MUST включаться в структуру компонентов.
- Что если Figma API вернёт неизвестный тип ноды?
  MUST залогировать предупреждение с node ID и типом, но продолжить
  парсинг остальных нод.
- Что происходит если output-директория уже существует?
  Система MUST перезаписывать файлы без предупреждения (детерминированное
  поведение). Старые файлы, не относящиеся к текущему запуску,
  НЕ удаляются — перезаписываются только файлы с совпадающими именами.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST извлекать все solid-цвета из fills
  и strokes каждой ноды. Градиенты (линейные, радиальные, угловые,
  diamond) извлекаются отдельным экстрактором — см. FR-001b.
- **FR-001b**: Система MUST извлекать все градиенты из fills:
  LINEAR_GRADIENT, RADIAL_GRADIENT, ANGULAR_GRADIENT, DIAMOND_GRADIENT
  — с полным набором gradientStops и handle_positions.
- **FR-015**: Система MUST обрабатывать IMAGE-заливки: сохранять
  imageRef и scaleMode в JSON, а также скачивать сами изображения
  через Figma Images API (`GET /v1/images/:key`) в поддиректорию
  `images/` внутри output-директории. Имя файла — `{sanitized_node_id}.png`,
  где символ `:` в node_id заменяется на `-` (напр. "1:23" → "1-23.png")
  для совместимости с Windows.
- **FR-002**: Система MUST извлекать типографику из TEXT-нод:
  fontFamily, fontSize, fontWeight, lineHeight, letterSpacing,
  textCase, textDecoration.
- **FR-003**: Система MUST извлекать spacing-значения из auto-layout
  нод: padding (4 стороны), itemSpacing, counterAxisSpacing.
- **FR-004**: Система MUST извлекать border-radius (единый и по
  углам — rectangleCornerRadii).
- **FR-005**: Система MUST извлекать эффекты: drop shadow, inner
  shadow, blur — с полными параметрами.
- **FR-006**: Система MUST извлекать данные компонентов: ID, имя,
  размеры, auto-layout настройки, описание, варианты.
- **FR-007**: Система MUST генерировать CSS Custom Properties файл
  с токенами, сгруппированными по категории (цвета, типографика,
  отступы, радиусы, тени). Градиенты НЕ включаются в CSS (сложные
  multi-stop значения не подходят для Custom Properties) — они
  доступны только в JSON (gradients.json).
- **FR-008**: Система MUST генерировать JSON-файлы для каждой
  категории токенов (colors.json, typography.json, spacing.json
  и т.д.).
- **FR-009**: Система MUST принимать Figma file ID и Personal Access
  Token как входные параметры.
- **FR-010**: Система MUST поддерживать фильтрацию по конкретной
  странице или node ID.
- **FR-011**: Система MUST генерировать AI-контекст файл с правилами
  использования токенов, топ-значениями по usage, и маппингом
  компонентов.
- **FR-012**: Система MUST нормализовать имена свойств в единый
  формат (snake_case для JSON, kebab-case для CSS).
- **FR-016**: Система MUST именовать токены по следующей стратегии:
  если у элемента есть Figma Named Style — использовать имя стиля
  (нормализованное в kebab-case); если стиля нет — авто-генерировать
  имя: цвета — по hue/luminance, типографика — по family-size-weight,
  spacing — `spacing-{value}` (напр. `spacing-16`), radius —
  `radius-{value}`, shadows — `shadow-{type}-{index}` (напр.
  `shadow-drop-1`), gradients — `gradient-{type}-{index}` (напр.
  `gradient-linear-1`).
- **FR-019**: Метрика usage_count для токенов MUST считаться как
  количество уникальных нод, содержащих данное значение (не количество
  fills/strokes внутри одной ноды).
- **FR-013**: Система MUST сохранять node_id каждого элемента для
  трассировки до исходного Figma-файла.
- **FR-014**: Система MUST выводить понятные сообщения об ошибках
  при проблемах с API (невалидный токен, несуществующий файл,
  rate limit).
- **FR-017**: При получении HTTP 429 (rate limit) система MUST
  автоматически ждать количество секунд из заголовка `Retry-After`
  и повторять запрос. Максимум 3 повторных попытки, после чего —
  завершение с ошибкой и сообщением о rate limit.
- **FR-018**: Система MUST принимать API-токен через аргумент
  `--token` или переменную окружения `$FIGMA_TOKEN`. Токен MUST
  маскироваться в любом логе/выводе (формат: `figd_***`). Токен
  MUST NEVER включаться в выходные файлы (JSON, CSS, Markdown).

### Key Entities

- **FigmaFile**: Представляет Figma-файл — ID, имя, дерево
  документа, метаданные (lastModified, version).
- **DesignToken**: Обобщённый тип извлечённого токена — union из
  ColorToken, GradientToken, TypographyToken, SpacingToken, RadiusToken,
  ShadowToken, ImageToken. Каждый подтип содержит: имя, значение,
  CSS-представление (где применимо), usage_count и node_id.
  Детальная структура — см. data-model.md.
- **Component**: Figma-компонент — ID, имя, размеры, layout-свойства,
  варианты, дочерние ноды.
- **ParseContext**: Контекст парсинга — target file ID, token, фильтры
  (page/node), опции (include-hidden, output format).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Извлечённые цветовые значения MUST совпадать с Figma
  с точностью до 1/255 на канал (допустима потеря при округлении RGBA).
- **SC-002**: Извлечённые размеры шрифтов, line-height, letter-spacing
  MUST совпадать с Figma с точностью до 0.5px.
- **SC-003**: Все компоненты типа COMPONENT и COMPONENT_SET из
  Figma-файла MUST присутствовать в выходных данных (0% потерь).
- **SC-004**: Парсинг файла с 5000 нод MUST завершаться менее чем
  за 30 секунд (без учёта времени ответа Figma API).
- **SC-005**: Сгенерированный CSS-файл MUST быть валидным CSS
  (проходить CSS-валидатор без ошибок).
- **SC-006**: AI-ассистент, используя сгенерированный контекст,
  SHOULD воспроизводить компонент с визуальным соответствием оригиналу
  не менее 90% (по субъективной оценке разработчика).

## Clarifications

### Session 2026-02-27

- Q: Как обрабатывать IMAGE-заливки (фоновые изображения, аватары, иконки)? → A: Сохранять ссылку (imageRef, scaleMode) И скачивать картинки через Figma Images API в output-директорию.
- Q: Как именовать токены в выходных файлах? → A: Приоритет Figma Named Styles (если у элемента задан именованный стиль), fallback на авто-генерацию по hue/luminance.
- Q: Как вести себя при rate limit (429) от Figma API? → A: Авто-retry — ждать Retry-After секунд, повторить до 3 раз, после чего завершиться с ошибкой.
- Q: Как обращаться с Figma API токеном (безопасность)? → A: Принимать через --token или $FIGMA_TOKEN env, маскировать в логах (figd_***), никогда не включать в output-файлы.

## Assumptions

- Пользователь имеет Figma Personal Access Token с правами чтения
  файла.
- Целевые Figma-файлы используют Figma REST API v1 (актуальная версия).
- Инструмент работает как CLI-утилита (Node.js), запускаемая вручную
  или интегрируемая в CI/scripts.
- Формат выхода по умолчанию: JSON + CSS. Дополнительные форматы
  (SCSS, Tailwind config) — вне scope этой версии, но архитектура
  MUST позволять добавить их позже.
- Поддержка MCP-протокола — вне scope v1. Текущий фокус — standalone
  CLI-инструмент, который может быть обёрнут в MCP-сервер позже.
